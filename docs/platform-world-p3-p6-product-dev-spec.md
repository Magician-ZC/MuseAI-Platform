- 本文的唯一先验假设是：用户可能愿意持续回来查看“自己的角色离开自己后发生了什么”。在证明该行为前，世界大厅、钱包和直播都不是前置条件。

# 第一部分 · 产品文档

## 1. 总体定位与形态演进

### 1.1 双模式客户端（Codex 型形态）

现有 Tauri 桌面客户端升级为**双模式**，参照 Codex / WorkBuddy 类客户端的形态——桌面应用 + 账号登录 + 云端能力解锁：

| 模式 | 登录要求 | 能力 | 数据位置 |
|---|---|---|---|
| **本地模式** | 不需要 | P0–P2 全部能力：角色提取、DNA 卡、知识包、单机自主叙事，自带 API Key | 本地 `~/Documents/MuseAI/` |
| **平台模式** | 需要 | 已获批的增量能力；首期为角色版本发布、精选放置世界、日报与轻干预 | 云端 |

**Local-first 承诺（产品红线）**：本地能力永不锁登录、永不因平台不可用而降级；登录只解锁增量，不劫持存量。这里的 local-first 指数据与宿主控制权，不等于远程模型调用时“文本不离开设备”。社区规模不作为本文未经实时核验的论据。

### 1.2 一句话定位（平台侧）

> 把你亲手养出的角色，投进一个有别人角色的世界，看它替你活、替你斗、替你结缘——你是它的命运，不是它的骑手。

### 1.3 平台世界与三方案的关系

前期讨论的三个方案保留为三种时间制式，但不同时产品化。先只验证放置房；章节房与赛事房在上一阶段达门后再做需求发现：

| 房型 | 对应方案 | 时间制式 | 一局时长 | 情感契约 | 开办者 | 交付期 |
|---|---|---|---|---|---|---|
| **放置房** | 方案 3（房主自建） | 放置制：每日 2–4 节拍 | 数周–数月 | 牵挂与惊喜 | 房主（官方模板） | P4 |
| **章节房（条件性）** | 方案 2（无限流） | 章节制：会话 30–60 分钟 + 离线夹层 | 数天–数周 | 成长与收集 | 官方 | P5 期权 |
| **赛事房（条件性）** | 方案 1（直播竞技） | 赛事制：回合 5–10 分钟 | 单场 2–3 小时 | 竞技与围观 | 官方 + 签约主播 | P6 期权 |

变速时间（EVE 模型：平时放置蓄势、赛季开打高潮局）作为 P6 之后的融合形态，本期只预留数据接口（放置房角色可报名赛事房）。

## 2. 平台世界总体设计

### 2.1 首个用户闭环与世界大厅

- **P4a 不建设空大厅**：先由官方提供 2–3 个精选放置世界，用户通过邀请链接或“把角色投进世界”直接进入；供给、审核和活跃密度可控。
- 只有活跃世界数量、加入转化和 7 日回访达到 P4a 门槛后，P4b 才建设大厅筛选与公开发现。
- P4b 房型选择器先只显示放置房（筛选：题材 / 节拍速度 / 开放状态）；P5/P6 真正上线后再出现相应入口，不展示未来空能力。
- 我的世界（已投放角色的世界列表，未读日报 / 战报角标）
- 世界详情页：世界书简介、规则摘要、当前角色阵容（公开部分）、边界/费用/奖励（如适用）、**合规信息展示**（AI 生成内容标识、世界规则与仲裁公开承诺）
- 日报是日回访产品，P4a 必须同时交付响应式 Web/PWA 阅读页和用户明确订阅的通知渠道；现有“同一局域网移动端”不能承担离家后的召回。

### 2.2 账号系统

- **注册与年龄分层**：P4a 使用手机号/邮箱验证码与年龄声明；是否需要全量实名认证，由网络游戏、支付、拟人化互动和属地合规结论决定，不在注册第一天预设为所有用户的硬要求。需要验证时只保存第三方返回的状态、referenceId 与必要审计信息，不保存原始身份证影像。
- **未成年人默认保护**：在无法可靠判断年龄前按保守策略限制付费与高风险内容；如构成网络游戏，接入相应实名防沉迷要求。家长同意、时长提醒、内容分级与退款通道在适用时进入发布阻断门。
- **账号成长（P5/P6 条件性）**：P4a 不做。若后续验证需要，经验可来自参与节拍、创作贡献或赛事荣誉，但不得用日常打卡和付费刺激替代角色故事本身的回访价值。
- **等级收益边界**：P4a 不做账号等级。若后续引入，公开或竞技世界必须统一控制预算，不能让等级兑换托梦次数、装备栏或入场情报等实质优势；优先奖励外观、归档容量和创作工具。任何影响世界结果的付费项须单独做公平性与合规评审。

### 2.3 角色资产上云

沿用 P0 的三分离原则并延伸到云端：

```text
本地模板（私有，永远归用户）
  → 用户显式「发布某一版本」 → 不可变云端快照（进审核队列：机审 + 风险分层人审）
    → 审核通过 → 可投放世界
      → 世界内产生运行状态（云端，随世界存续）
        → 运行状态不回写模板；用户可在世界结束后领取「经历档案」归档到本地羁绊系统
```

- **权利与审核不因私密房消失**：所有上传云端的角色卡、原文片段、知识包和世界模板都要声明权利基础与允许用途，并接受最低安全检查。私密房只降低发现与传播范围，不能用“用户自担责”替代平台的数据、内容和版权义务；公开分发再叠加更严格的权利证明和人审。
- **最小发布清单**：默认只上传运行所需的角色版本与权利元数据，不上传本地证据账本、原文引文、用户关系记忆或其他故事历史；确需上传的知识片段逐包授权并在发布前预览。云端副本保存一份可审计 manifest，列明字段、用途、可见范围和删除策略。
- **版本与撤回**：世界只引用发布时的不可变角色快照，本地后续编辑不回写已运行世界。用户可撤下后续投放、导出或删除云端资产；对正在运行世界，按入场协议选择“立即退出 / 生成离场事件 / 世界结束后删除”，并清楚说明依法或履约必须保留的最小日志与期限。
- 同源角色多副本（五个孙悟空同场）作为特性支持：投放时生成记忆分叉标识，各副本独立运行状态。

### 2.4 干预系统（三环模型的平台化规格）

沿用已定稿的三环设计，此处规格化：

| 环 | 时机 | 操作 | 限制 | 付费属性 |
|---|---|---|---|---|
| 构筑环 | 入场前 / 节拍间 | 卡调校、知识包挂载、结盟预设 | P4a 免费且同额度 | 后续商业化再评审 |
| 影响环-托梦 | 世界运行中 | 给角色一条"心声 / 直觉 / 执念"（≤100 字） | 低优先层；角色可忽略；按世界设固定额度与冷却，竞技世界所有人相同 | P4a 不付费 |
| 影响环-道具 | 世界运行中 | 向角色 / 场景投放有限规则标签的道具与环境事件 | 道具改变局面不改变意志；P4a 仅免费测试 | P4b 前单独评审 |
| 边界与同意 | 入场前配置；高风险事件触发时复核 | 死亡、永久退场、严重伤害、关系/亲密边界、成人内容等 | 以入场协议为默认；真正不可逆或超出预授权范围时暂停并请求确认，不以超时默认同意 | 无 |

**托梦的技术语义**（衔接 P2 上下文硬序）：注入位置在「本轮检索片段」之后、「表达与文风要求」之前，并显式标注"这是一个外来的声音，你可以依自己的本性忽略它"。角色抗命是特性不是缺陷，抗命事件进日报高光。

同意记录只展示规则、选项和后果，不展示模型隐藏推理。多角色事件需要在入场协议中预先确定冲突规则；未响应时默认选择更可逆、更保守的结果，而不是默认同意死亡或永久关系变化。

### 2.5 三房型详细设计

**放置房（P4）**

- P4a 由官方创建 2–3 个精选世界，用户只需选角色、确认边界并投放；不先让普通用户承担建房与冷启动负担
- 每日 2–4 节拍由平台调度执行；角色间自主交互，导演 Agent 注入周期性世界事件
- **日报系统**：每日按角色主人视角生成《你的角色昨日人生》——高光事件卡 + 关系变化 + 一句角色独白；这是放置房的核心产品，不是附属品
- P4a 通过后，P4b 才开放私密开黑房与有限模板创建；公开 UGC 世界、模板市场和分成不进入首个付费版本
- 用户可暂停通知、暂停角色、撤回角色、查看“为什么发生”与申诉；日报必须明确区分公开事实、角色私密视角和模型推断

**章节房（P5）**

- 官方制作世界（无限流结构）：主线任务、隐藏道具、结局分支
- 门票入场；章节 = 一次 30–60 分钟主动会话（玩家实时托梦 / 投道具，节奏比放置房密）
- 章节间离线夹层：角色自动训练 / 探索 / 结社，回来领离线收益（AFK 结构）
- **跨世界背包**：通关可携出道具 / 称号进入账号背包，在其他章节房使用；道具跨世界效果由目标世界规则重映射（叙事数值化表，见 §9.5）
- **主场优劣势机制**：本书角色进本书世界——优势 = 原作剧情作为预知型知识包自动挂载；劣势 = 原作宿命作为大纲硬节点压制（世界线修正力）。P1 知识包与 P2 约束模型原生实现
- **动态装配与实例分流**：世界模板与世界实例分离——同一部小说的副本达到人数上限即自动开设下一实例；每个实例在开团时由「开局装配器」读取全体入场角色的结构化指标（执念、恐惧、剧情种子、拒绝规则、关系网、来源体系），从**预审核内容池**中选择并参数化隐藏道具、支线钩子与结局分支，使每个实例的副本内容因阵容而异。装配只做选择与参数化，不自由生成主线（机制见 §9.5）
- **物品准入策略**：每个世界模板声明准入规则（全开放 / 体系黑名单 / 白名单 + 强度上限），依据物品的来源体系标签在入场时由服务端校验——魔法世界可拒收能量武器与机械造物；被拒物品留在账号背包不随角色入场。玩家自建世界（P4b 私密房起）自由选择准入策略，官方世界由运营配置（见 §9.5）
- 合规门：章节房含账号成长、虚拟道具与互动玩法，**开发前即完成网络游戏定性法律评估，收费公测前完成适用许可**（见 §4）；免费内测不当然改变业务定性或免除内容、未成年人和数据义务

**赛事房（P6）**

- 依托抖音直播小玩法生态挂靠（不自建直播）：观众礼物经平台网关映射为场内道具 / 环境事件
- 主播 = DM + 解说，回合 5–10 分钟，主播在节拍间表演
- 唯一胜者赛制；**付费边界**：观众买"过程"不买"结果"——道具与复活赛资格可买，免死与最终判定不可买；胜者奖励为荣誉性（称号、立绘框、赛季榜）非强度性
- **透明战报**：每回合仲裁器输出可查战报（谁做了什么、判定依据、道具生效记录），对抗"是不是剧本"质疑
- 主播分成：礼物流水分成 + 赛事冠名

### 2.6 商业化阶段门与经济系统

- **P4a 无充值、无平台币、无分成**。先证明日报打开、7 日回访、角色撤回率、运营工时和单位角色成本。
- P4b 只有在用户愿付费验证与法律评估通过后，优先测试透明订阅或世界运行额度；不默认用平台币、礼物和复杂钱包。
- 如未来采用平台币：用户余额不可提现、不可转账、不可兑换随机或胜负结果；订单、退款、对账、限额和未成年人保护为发布阻断门。
- 创作者结算与用户钱包是两套账：前者只面向签约主体，涉及合同、税务、发票、KYC 和退款准备金；不能把“用户币不可提现”与“创作者可结算”混在一个 `wallet` 概念里。
- 分成比例在真实成本、渠道费、税费、退款与内容审核工时出来前保持 TBD，不在规格中预填 30/70。

### 2.7 呈现层决策：纯文字还是画面

结论：**不做实时 3D / 游戏引擎画面，也不停留在纯文字——采用「结构化事件流 + 分层渲染」架构**，四层渐进增强：

| 层 | 内容 | 实现 | 边际成本 | 适用 |
|---|---|---|---|---|
| **L0 文字流** | 叙事正文、对话 | 引擎直出 | 低 | 全房型阅读态、日报正文 |
| **L1 结构化卡片** | 事件卡、关系图谱、势力地图、角色状态面板、战报时间轴 | 前端组件渲染（React + Canvas），引擎输出结构化事件即可驱动 | 低 | 全房型默认界面；优先投入 |
| **L2 静态插画** | 角色立绘、章节场景图 | AI 生图管线，立绘带风格锁定 | 待真实模型、审核与存储成本验证 | P4b 候选；P4a 不作为上线依赖 |
| **L3 高光切片** | TTS 配音 + 条漫式动态分镜（motion comic），可选 AI 视频 | 切片合成管线，仅对高光事件按需生成 | 中（按需触发） | 赛事房直播素材、传播分发 |

- 关键架构决策：**叙事引擎与呈现层解耦**——引擎输出 `DomainEvent + StatePatch`，平台服务再生成经过受众投影与审核的 `WorldEvent`（见 §9.4）。呈现层永远不直接消费私有原始事件。
- 最低呈现配置：P4a 放置房只要求 L0+L1；日报 L2 配图先做按需实验。章节房与赛事房的呈现要求只有在对应阶段获批后才冻结。
- 文本、图片、音频、视频的界面展示、复制、下载与导出都要遵循当时适用的显式/隐式 AI 内容标识规则；不能只在世界详情页放一个总标签。
- 明确排除：实时 3D、预制美术资产库、虚拟人直播形象（与 AI 叙事的开放性冲突，工程量对团队不可行）。

## 3. 后台管理系统（产品视角）

独立 Web 应用（运营与管理员使用），八大模块：

| 模块 | 核心功能 | 期 |
|---|---|---|
| 用户管理 | 账号检索、年龄/身份验证状态（不展示原始证件）、封禁 / 解封、设备与风控标记、数据导出/删除工单 | P3 |
| 内容审核 | 角色卡发布审核队列（机审结果 + 人审操作）、世界模板审核、举报处理、敏感词库维护 | P3 |
| 世界运营 | P4a：活跃世界监控、卡死诊断、暂停/重推/回滚、官方放置世界配置；章节工具与赛事日历随 P5/P6 增加 | P4a 起 |
| 经济运营 | P4a 不建设；P4b 获批后增加订单、退款、对账；签约创作者结算与用户余额分账管理 | P4b 条件性 |
| 数据看板 | 激活、日报送达/打开、次日/7 日回访、撤回率、人工运营分钟/世界日、token 与人民币成本、质量/安全告警 | P3 起逐步 |
| 模型与 Prompt 治理 | 模型路由、Prompt 版本/灰度/回滚、效果抽检；世界钉住版本 | P4a |
| 风控 | P4a：批量注册、越权访问、角色卡注入与内容风险；支付/礼物风控随 P4b/P6 增加 | P4a 起 |

## 4. 合规与运营门槛（阶段门）

| 事项 | 触发点 | 所在门 |

## 5. 平台轨道非目标

- 不做实时 3D 与游戏引擎渲染；不做虚拟人直播形象
- 不做角色卡交易市场、不做任何可兑现 / 可转让资产（模板市场为内容授权使用，非资产转让）
- 不自建直播基础设施，赛事房只挂靠成熟直播平台
- 不做付费购买胜负；不做免死道具
- 版号问题未明确前，不对章节房收费公测
- 不为登录用户关停或降级任何本地能力
- P4a 不做充值、平台币、分成、公开 UGC 大厅、模板市场、账号等级或付费干预

## 6. 平台侧指标

- **P4a 北极星**：收到首份日报的用户中，7 日内至少 3 次主动打开日报或世界详情的比例。
- 激活漏斗：发布角色快照 → 完成边界协议 → 投放成功 → 首个 tick 成功 → 日报送达 → 日报打开。
- 用户价值：次日/7 日回访、日报有效阅读、用户主动点“为什么发生”、继续投放/邀请、角色撤回率与原因；不把托梦点击本身当价值。
- 质量与安全：角色主动剧情贡献、趋同告警、私密信息泄漏、被阻断事件、边界确认/申诉、生成内容举报率。
- 工程与运营：tick 成功率、重复事件率、p95 日报到达时延、每世界日成本、每世界日人工运营分钟、审核积压与恢复时间。
- 商业化门（P4b）：先验证明确价格下的订阅/额度转化和毛利，再建设支付；成本收入比只在收费后使用，不能在 P4a 以假收入计算。
- P5/P6 指标只有在对应期权启动后定义；当前不以前景指标倒逼提前建设章节/直播系统。

**技术选型与理由**：

| 选型 | 理由 |
|---|---|
| 后端 Rust axum | 团队栈连续（`mobile_server.rs` 已用 axum）；narrative-engine 以 crate 共享，桌面与云端不写两遍 |
| 模块化单体（非微服务） | 小团队现实；模块边界按上图划分，未来可拆 |
| PostgreSQL + Redis | 事件流与账本用 PG（JSONB 存 WorldEvent）；Redis 做 tick 队列、会话、限流 |
| 管理后台 React 19 + antd v6 | 与客户端同栈，组件复用（战报卡、关系图组件两端共用） |
| 部署国内云 + 可审计的基础设施即代码 | 地域由备案、数据与供应商要求决定；Compose 可用于原型，外部 Alpha 前补备份、密钥、监控与恢复演练 |
| 模型接入 | 优先接入可满足适用备案/登记与数据要求的模型；上游合规不能替代应用侧内容、标识、投诉和数据责任 |

**架构核心决策：先保持宿主无关，再决定抽包路径**。P0–P2 已要求通过 trait 隔离宿主能力。P3 先写 characterization tests 和 crate 边界 ADR，再将相关模块抽为 workspace crate；若抽包会阻断 P4a 验证，可先由云端 worker 通过版本化协议调用独立运行器。验收不是“代码移动成功”，而是桌面行为一致、状态兼容且可回滚。

## 9. 后端服务设计

### 9.1 核心 API 草案（REST + WebSocket）

```text
POST /auth/challenge, /auth/login, /auth/refresh, /auth/logout
POST /identity/verification            # 仅适用时；保存 provider reference/status
POST /assets/characters                # Idempotency-Key；发布不可变版本 → 审核
GET  /assets/characters/:id/status
POST /assets/characters/:id/withdraw   # 停止后续投放，运行中按入场协议处理
DELETE /assets/characters/:id          # 返回删除范围、保留项和完成状态
GET  /worlds                           # P4a 仅精选放置世界；cursor 分页
POST /worlds/:id/join                  # Idempotency-Key + 角色版本 + 边界协议
POST /worlds/:id/leave
GET  /worlds/:id/events?cursor=        # 只返回当前 principal 的投影
WS   /worlds/:id/stream                # 带 audience/lastEventId，支持重连补偿
POST /worlds/:id/interventions         # Idempotency-Key + expectedWorldRevision
POST /worlds/:id/consents/:id/respond  # 明确同意/拒绝；重复请求幂等
GET  /me/reports?cursor=&date=
GET  /me/export, DELETE /me            # 异步任务 + 状态查询
PUT  /me/notification-preferences
# P4b 条件性：/billing/orders, /billing/refunds, /billing/balance
```

所有会产生副作用的接口必须支持 idempotency key、鉴权主体、schema 版本和并发 revision；列表使用 cursor 分页。错误返回稳定 error code，不把模型或数据库内部信息暴露给客户端。P5/P6 API 不提前冻结，避免让未验证玩法污染 P4a 核心域。

### 9.2 世界运行时（runtime）

- **tick 调度器**：按房型节拍配置生成 tick 任务入 Redis 队列；worker 池消费，每 tick = 一次 P2 回合循环（导演设局 → 并发 role_decide → 仲裁 → 场景写作 → 审校 → 状态提交 → 事件流落库）
- **幂等与恢复**：tick 有唯一 id，读取 `baseRevision`，在同一事务中写 StatePatch、DomainEvent、outbox 与新 revision；worker 崩溃后可安全重试，不产生重复事件、重复通知或重复计费
- **预算熔断**：P4a 按世界日 token/人民币双预算；超限先暂停并告知运营与用户，不悄悄降低模型导致人格漂移。P4b 收费后才加入余额预检
- **卡死干预**：单 tick 超时（模型侧异常）自动重试一次 → 失败则标记世界「暂停」，进后台世界运营队列人工处理
- **版本钉住**：每个世界固定 `engineVersion/promptSetVersion/modelRouteVersion/worldTemplateVersion`；升级通过显式迁移和 canary world，不让存续世界在半途静默变更行为
- **通知 outbox**：日报发布与通知发送解耦；退订、静默时段、失败重试、去重和送达状态可审计
- P6 条件性：若赛事房独立立项，tick 再增加主播手动/半自动触发与 livegate 输入；相关代码不进入 P4a runtime

### 9.4 DomainEvent 与 WorldEvent（引擎、平台与呈现的契约）

```typescript
type DomainEventType = 'action_resolved' | 'dialogue_spoken' | 'relation_changed'
  | 'resource_changed' | 'outline_progressed' | 'consent_requested';

interface DomainEvent {               // narrative-engine 产物，不含云端账号/审核/呈现字段
  schemaVersion: 1;
  id: string;
  runId: string;
  sequence: number;
  type: DomainEventType;
  actorIds: string[];
  targetIds?: string[];
  fact: Record<string, unknown>;       // 实现时按 type 使用判别联合与独立 JSON schema
  statePatchId: string;
  causedBy: string[];
}

interface WorldEvent {
  schemaVersion: 1;
  id: string;
  worldId: string;
  tick: number;
  sequence: number;
  domainEventId: string;
  engineVersion: string;
  promptSetVersion: string;
  type: 'action' | 'dialogue' | 'conflict' | 'alliance' | 'item'
      | 'status' | 'arbiter' | 'world' | 'consent_request';
  actors: string[];
  targets?: string[];
  publicProjection?: { summary: string; narrative?: string; quote?: string };
  privateProjections?: Array<{
    audiencePrincipalIds: string[];
    summary: string;
    narrative?: string;
  }>;
  visibility: 'public' | 'restricted' | 'private';
  audiencePrincipalIds?: string[];     // 非 public 必填；查询层和推送层双重校验
  impactSummary: Array<{ characterId: string; fields: string[] }>;
  decisionSummary?: string;            // 面向用户的规则依据，不含隐藏推理/私密 prompt
  ruleRefs?: string[];
  moderation: 'pending' | 'approved' | 'quarantined' | 'rejected';
  aiLabel: { visible: boolean; metadataRef?: string };
  occurredAt: number;
  assets?: { portraitId?: string; sceneImageId?: string; clipId?: string };
}
```

`StatePatch` 是状态变化的唯一事实源，`WorldEvent` 只做经过受众投影、内容审核和 AI 标识后的展示。不得使用可执行的 `field/delta: string` 让呈现事件反向修改状态。`DomainEvent.fact` 的 `Record` 只是文档缩写，实现必须按 type 建判别联合和 schema registry。私有投影与公共投影分开存储/缓存，数据库查询、WebSocket fan-out 和缓存键都强制带 principal；不能依赖前端隐藏。透明战报只展示规则、输入事实与结果摘要，不展示 chain-of-thought、其他角色私有状态或安全规则细节。

### 9.5 章节房规则系统（P5 条件性设计，不进入 P3/P4a；本节为机制规格，API 不提前冻结）

**A. 叙事数值化表**

跨世界道具需要「叙事效果 → 规则效果」的映射层：道具定义 = 叙事描述 + 结构化效果标签（`advantage:combat`、`reroll:once`、`info:reveal` 等有限枚举）。仲裁器只认标签，写手只用描述。标签枚举表由官方世界设计维护（后台可配），这把 MMO 级平衡问题收敛为有限标签集的调参问题。

**B. 物品来源体系与世界准入**

```typescript
interface ItemDefinition {
  id: string;
  narrative: string;               // 写手只用这段
  effectTags: string[];            // 仲裁器只认这组有限枚举（见 A）
  origin: {
    worldTemplateId: string;       // 产出世界
    cosmology: string[];           // 体系标签，官方维护的有限枚举：magic / tech / cultivation / mundane / ...
    powerTier: number;             // 强度层级 1–5
  };
}

interface WorldAdmissionPolicy {
  mode: 'open' | 'denylist' | 'allowlist';
  cosmologies?: string[];          // denylist / allowlist 的作用对象
  maxPowerTier?: number;
  rejectedHandling: 'stay_in_backpack'   // 拒收：留在账号背包，不随角色入场
                  | 'sealed_inside'      // 入场但封存：本世界内不可用
                  | 'translate';         // 规则转译：重映射为本世界等阶等价物（官方世界可选）
}
```

- 体系标签是官方维护的有限枚举（后台可配），不接受自由文本，避免标签失控。
- `translate` 模式（能量武器进魔法世界被转译为等阶魔导器）只重写叙事外皮，`effectTags` 与 `powerTier` 不变或降档，保证转译不成为强度后门。
- **准入执行为服务端双重校验**（衔接 §9.6）：入场时（join 请求携带的物品清单按策略过滤）+ 运行时（仲裁器拒绝任何未准入物品的使用行动并记 `risk_event`）。客户端展示准入结果，但不参与判定。
- 玩家自建世界自由选择策略，提供预设：全开放 / 仅同体系 / 严格白名单。

**C. 副本模板与开局装配（roster-conditioned assembly）**

回答「副本内容能否随入场角色动态变化」：能，且这是 LLM 叙事相对传统游戏的独有优势位——传统游戏的动态导演止步于节奏级（Left 4 Dead 式 AI Director）与布局级（roguelike 程序化生成），LLM 可以做到内容级。机制：

- **模板 / 实例分离**：世界模板 = 静态骨架（主线硬节点序列、结局池、隐藏内容池、支线钩子池——全部预审核）+ 装配规则。同一模板可开任意多实例，报名满员或开团时创建新实例。
- **装配输入**：全体入场角色的结构化指标——`dramaticCore.coreFear / deniedDesire`、`agency.plotSeeds / refusalRules`、关系网、`origin.cosmology`、主场标记。P0 的 DNA 字段天生就是装配器的输入接口；§2.5 的主场优劣势机制是本机制的一个特例。
- **装配动作**（实例创建时一次性执行，数次模型调用 + 规则选择，不进 tick 循环）：
  1. per-character 钩子：每个入场角色至少绑定 1 个与其执念 / 恐惧 / 剧情种子挂钩的隐藏内容（从池中选择并参数化）；
  2. 结局分支条件化：结局池按阵容特征加权启用（全员谋略型 → 智斗线结局权重上调）；
  3. 阵容级参数：支线权重、冲突密度、资源稀缺度按阵容画像调整。
- **安全与合规边界**：装配只做「从预审核池中选择 + 参数化」，不自由生成新主线；装配产生的连接文本过机审后才生效；装配结果固化为该实例的 `worldTemplateVersion` 派生版本并钉住（衔接 §9.2 版本钉住），实例存续期内不静默变更。
- **公平性**：个性化内容附带难度分标注；实例若含竞争性结算，难度分进入配平（开放问题 §15.9）。

### 9.6 服务端权威与客户端零信任（P3 起全局原则，非条件性）

平台世界的一切状态转换只发生在服务端；客户端是「意图输入 + 投影展示」终端，不是数据源：

- **客户端只能提交意图**（发布、投放、托梦、道具使用、同意响应、订单），每个意图携带鉴权主体、Idempotency-Key 与 `expectedWorldRevision`（§9.1 已规定）。服务端依据**自己持有的状态**校验意图合法性：角色是否真的在场、道具是否真的在其背包、额度是否足够、物品是否满足目标世界准入。非法意图拒绝并记 `risk_event`。
- **客户端上报的任何数值一律不作为事实源**。背包、道具、经历、关系等资产的取得只有两条服务端写入路径：tick 事务内经 reducer 校验的 `StatePatch`，或支付履约（P4b）。不存在「客户端声明我拥有」的写入接口——抓包篡改请求体最多得到一个被拒绝的意图与一条风控记录，不可能让背包里凭空多出装备。
- **读写通道分离**：`WorldEvent` 投影是纯只读展示层（§9.4 已禁止可执行 delta）；不允许存在任何以事件回放 / 回传方式修改状态的接口。
- **传输与重放**：全链 TLS；WS 订阅带 principal 校验；副作用接口的幂等键 + revision 检查使重放攻击退化为无效重复。
- **边界说明**：本地模式（P0–P2 单机）数据本就归用户，修改自己的存档不构成作弊；服务端权威只约束平台世界。本地资产进入平台的唯一通道是「发布不可变快照 → 审核」，本地侧的后续修改不影响已发布版本（§2.3 已规定）。

## 10. 后台管理系统（技术）

- 独立 React + antd Web 应用，P3 先做超管 / 运营 / 审核 / 客服最小 RBAC；财务角色在 P4b 增加，全操作留痕审计
- 审核工作台：机审结果预标注（内容安全 API + 注入检测规则）+ 人审快捷键流水线；角色卡审核展示「卡片全文 + 机审命中点 + 同作者历史」
- **Prompt 治理**（LLM 产品的运维核心）：线上各场景 prompt 版本化存储、diff 视图、按世界灰度发布、一键回滚；效果抽检 = 定期抽样 tick 的输入输出对，人工标注质量分入看板
- 成本看板：按世界 / 房型 / 模型聚合 token 消耗与人民币成本，成本收入比排序，熔断记录
- 世界运营工作台：暂停世界的脱敏诊断视图（调用元数据、规则命中、错误码、用户授权后才查看必要内容）、手动重推 / 跳过 / 回滚到快照；高权限查看有理由填写、双人授权和审计日志

## 11. 呈现层技术

- **L1 组件库**（客户端与后台共用）：事件卡、时间轴、关系图谱（力导向图，数据源 = relations 状态）、势力地图（世界模板定义地点拓扑，角色位置标注）、状态面板
- **L2 生图管线**：P4b 条件性。立绘与场景图按需异步生成，缓存期限服从资产撤回、删除和模型供应商条款；不承诺“永久缓存”，失败降级为无图
- **L3 切片管线（P6 条件性）**：高光判定 → TTS 配音 + 条漫分镜合成 → 输出 MP4 / GIF；P4a 不实现
- **观战与直播（P6 条件性）**：网页观战席、OBS 浏览器源和主播控制台在 P6 立项后再定接口

## 12. 分期计划

| 期 | 范围 | 验收门 | 合规门 |
|---|---|---|---|
| **PX 礼宾验证** | 不建平台；人工/半自动让 15–30 名 P2 用户把角色放进 2 个精选世界，连续 7 天发送日报原型 | ≥40% 收件用户在 7 日内主动打开 ≥3 次；有人主动追问/干预/邀请；运营工时与成本有记录。未通过则停止平台投入 | 使用已授权数据；明确测试协议与删除机制 |
| **P3 账号与云基座** | 宿主无关引擎边界与可选抽包；auth / 资产版本 / 审核 / 删除导出 / 事件投影 / 通知；桌面云通道 + 响应式 Web/PWA；管理后台最小用户/审核/看板 | 本地回归全绿；发布→审核→投放→撤回/删除链路；受众隔离与幂等测试；通知可退订；灾难恢复演练 | 外部测试前完成实际业务形态的备案、标识、未成年人、隐私与拟人化互动适用性判断 |
| **P4a 放置房免费封闭 Alpha** | 官方精选世界；runtime/tick/预算；日报 L0+L1；固定额度托梦；边界协议；运营工具。不做钱包/大厅/公开 UGC/L2 依赖 | 先 7 日可靠性测试，再做 30 日 soak；无未计划工程干预，人工审核与运营正常记录；达到北极星、质量、安全与成本门 | 外部 Alpha 所需事项完成；内容审核与投诉/删除值班可用 |
| **P4b 付费 Beta（条件性）** | 透明订阅/运行额度优先；订单、退款、对账；私密建房；是否加入平台币/道具/L2 由实验决定 | 明确价格下转化、毛利与退款可接受；支付/账本幂等；不降低角色公平和信任 | 支付、税务、未成年人、消费者权益与必要许可完成 |
| **P5 章节房（期权）** | 先需求发现；获批后再做官方世界工具链、章节会话、离线夹层和有限跨世界物品 | P4b 留存与单位经济成立，且章节原型证明完读与二世界转化 | **网络游戏定性/版号等专项法律评估**，结论未明确不对外收费 |
| **P6 赛事房（期权）** | 先用非付费、非实时原型验证观看性；获批后再做主播控制台、战报、切片与平台接入 | 与签约创作者的非付费试播证明观看、复看与公平信任；直播平台书面可接入 | 平台玩法审核、主播/版权/内容安全与付费合规完成 |

依赖关系：PX → P3 → P4a → P4b。P5 与 P6 不在固定关键路径上，分别由用户证据、法律结论与合作条件触发；任何一期未达门都应缩减、重做或停止，而不是靠下一期功能补救。

## 13. 成本模型与容量规划

- 单 tick 先以 `(活跃角色 N + 设局 + 仲裁 + 写作 + 审校/重试)` 建调用预算，但费用必须按各调用的真实输入/输出 Token、缓存折扣、重试率、模型单价与审核/图片/通知成本计算，不能只乘“平均调用价”。
- PX/P4a 记录每 tick、每活跃角色日和每位 7 日留存用户的 p50/p95 成本；“单角色日成本 < ¥0.5”在实测前只是待验证假设，不写为目标承诺。
- 容量门同时包含模型并发/限流、队列积压、PG 事务与事件表增长、通知高峰、审核吞吐和对象存储。先做压测与 30 日容量外推，再决定分区或横向扩容。
- P4b 定价公式至少包含模型与基础设施、内容审核、支付渠道、税费、退款/坏账、客服和创作者结算准备金；70% 只可作为待决目标毛利率，不是自动降频阈值。

## 14. 安全与风控

- **角色卡注入攻击（本产品特有威胁，最高优先级）**：UGC 卡片即 prompt，必须防御"操纵其他角色 / 越权获取秘密 / 劫持导演"类卡。措施：(1) 发布审核含注入模式静态检测（指令式语句、对系统 / 其他角色的直接命令句式）；(2) 决策上下文中他人卡内容仅以第三人称摘要形式出现，永不注入原文；(3) `role_decide` 输出 schema 强校验，越权字段丢弃；(4) 仲裁器对"读心 / 强制他人行动"类行动一律判无效并记 risk_event；(5) 后台风控看板聚类检出卡
- 支付风控（P4b 条件性）：充值限额、异常支付/礼物模式、未成年人拦截、退款与对账
- 数据：角色私有状态与私有事件按 visibility 硬隔离（查询层强制过滤，非前端过滤）；用户删除请求覆盖云端副本与运行数据，并清楚列出依法或为纠纷处理需限期保留的最小记录
- 内容安全采用分层管线：用户输入预检 → 结构化动作/状态不变量检查 → 面向用户的事件投影同步机审 → 高风险内容隔离并进入人审 → 低风险内容发布后抽检。原始 DomainEvent 不直接公开；高风险房间可暂停，低风险单条内容不必阻塞整个世界。
- 数据生命周期：给每类数据定义目的、可见主体、保存期、删除方式与备份清除期限；模型供应商日志与训练选项进入供应商评审。用户离开世界、撤回角色、注销账号和房主关闭世界均有明确状态机。
- 秘钥与权限：模型/审核/短信/支付凭据进入服务端 secret manager，管理员最小权限、MFA、关键操作双人复核；访问私有叙事内容必须有理由和审计。
- 事件安全测试覆盖 IDOR、WebSocket 越权订阅、缓存串租户、重放攻击、**客户端伪造状态与物品上报**（篡改意图请求体、伪造背包清单、绕过准入策略）、延迟任务在撤回后仍发布、提示词注入和日志泄密。

## 15. 开放问题

1. PX 的 15–30 名用户从 P2 哪一类高意向行为中招募？若没有可招募用户，平台假设不成立。
2. 日报通知首选渠道是什么，用户何时明确订阅、如何一键退订？无可靠触达时，放置房的日回访闭环不成立。
3. 用户投放的角色到底是“创作资产”还是“情感化代理”？该答案决定同意体验、拟人化互动适用性和文案边界。
4. 角色撤回时，正在进行的关系、共同行动和其他用户日报如何结算？需在 P4a 前定义离场状态机。
5. narrative-engine 的桌面/云端版本协商、世界版本钉住与迁移策略是什么？禁止强更本地能力来解决云端兼容。
6. 日报预生成还是打开时生成？P4a 做分层实验，同时测送达时延、浪费生成比例和成本，不提前定方案。
7. P4b 首个收费对象是订阅、角色运行额度还是私密房主？在 PX/P4a 访谈与价格实验前不建平台币。
8. 章节房和赛事房分别要看到哪些不可伪造的行为信号才启动 discovery？未写明即视为不立项。
9. 章节房实例装配的个性化强度与公平性边界：绑定角色执念的隐藏内容是否计入竞争性结算？难度配平规则在 P5 discovery 时与玩法一起定义。
